2 The Learning Loop (End-to-End)

The learning loop is the core feedback mechanism that makes SentCom self-improving. Every trade feeds back into the system, adjusting future confidence scores, model weights, and position sizing.

1. Data Ingestion
IB Gateway pushes live bars (1min → weekly) to MongoDB via ib_historical_data. 4 parallel collectors fetch historical data for ~2,800 symbols. 178M+ bars stored with sort("date", -1).limit(N) optimized queries.
▼
2. Feature Extraction
Vectorized NumPy feature engineering extracts 50+ features per bar: SMA/EMA ratios, RSI, Bollinger Bands, VWAP deviation, volume profiles, volatility metrics, regime features, and setup-specific patterns. Cached in feature_cache collection (~97% cache hit rate on subsequent runs).
▼
3. Model Training (13 Phases)
132+ models trained across XGBoost (GPU CUDA), PyTorch deep learning (VAE, TFT, CNN-LSTM), CNN chart pattern recognition, FinBERT NLP sentiment, and ensemble meta-learners. Walk-forward validation ensures models only promote if they beat existing accuracy.
▼
4. Setup Detection (Enhanced Scanner)
Real-time scanner monitors all active symbols for 20 setup types (10 long + 10 short): SCALP, ORB, GAP_AND_GO, VWAP, BREAKOUT, RANGE, MEAN_REVERSION, REVERSAL, TREND_CONTINUATION, MOMENTUM — plus their short equivalents. Each detected setup generates a LiveAlert.
▼
5. Confidence Gate (11 Layers)
Every alert passes through the additive scoring gate. Starts at 0 points, earns confirmation from regime alignment, model consensus, live prediction, CNN patterns, TFT temporal signals, VAE regime detection, and learning loop feedback. Decision: GO (≥55pts) | REDUCE (≥30pts, 60% size) | SKIP (<30pts).
▼
6. Trade Execution
GO/REDUCE decisions route to the TradeExecution engine, which sizes positions based on confidence score, risk-of-ruin calculations, and dynamic risk parameters. Orders submitted to IB Gateway. Real-time P&L tracked.
▼
7. Outcome Recording & Learning
The LearningLoopService captures every trade outcome (W/L, R-multiple, execution quality), updates per-setup win rates, detects tilt states, checks for edge decay, and generates calibration recommendations. This data feeds back into Layer 7 (Learning Loop Feedback) of the confidence gate, closing the loop.
▼
8. Re-Training
Periodic re-training incorporates new market data and trade outcomes. Models that improve on validation accuracy replace existing ones. Models that regress are kept at their previous version (accuracy gating). The cycle continues.

4 Model Inventory — Every Model Explained

4.1 XGBoost Models (GPU CUDA)

The backbone of the prediction system. XGBoost models are trained with CUDA GPU acceleration on the DGX Spark, using walk-forward validation (train on historical, validate on recent data).

Generic Directional Predictor XGBoost

Phase 1 | 7 models (1min, 5min, 15min, 30min, 1hr, daily, weekly) | File: timeseries_gbm.py

Foundation model. Takes 50+ technical features for ANY symbol and predicts whether the next N bars will move up, down, or flat. Trained on the entire universe (~2,800 symbols × all available history). Each timeframe has a different forecast horizon (e.g., 30 bars for 1min, 5 bars for daily).

Features: SMA/EMA ratios (5/10/20/50), RSI(14), Bollinger Band width, VWAP deviation, volume ratio, price momentum (5/10/20), ATR ratio, bar patterns, and regime indicators.

Setup-Specific Predictors (Long) XGBoost

Phase 2 | 17 models | File: setup_features.py, setup_training_config.py

Specialized models trained only on symbols/bars where a specific setup type was detected. Each setup has custom feature extractors:

  • SCALP — micro-structure features (tick volume, spread, momentum burst)
  • ORB — opening range features (range width, time-of-day, gap context)
  • GAP_AND_GO — gap size, volume confirmation, pre-market context
  • VWAP — VWAP deviation, reclaim probability, anchored VWAP
  • BREAKOUT — consolidation tightness, volume expansion, prior resistance
  • RANGE — range boundaries, mean-reversion tendency, support/resistance
  • MEAN_REVERSION — Z-score extremity, Bollinger squeeze, RSI divergence
  • REVERSAL — exhaustion signals, volume climax, trend fatigue
  • TREND_CONTINUATION — trend strength, pullback depth, moving average alignment
  • MOMENTUM — rate-of-change, relative strength, sector momentum

Setup-Specific Predictors (Short) XGBoost

Phase 2.5 | 17 models | File: short_setup_features.py

Mirror of the long setup models with inverted feature engineering. SHORT_SCALP, SHORT_ORB, SHORT_GAP_FADE, SHORT_VWAP, SHORT_BREAKDOWN, SHORT_RANGE, SHORT_MEAN_REVERSION, SHORT_REVERSAL, SHORT_MOMENTUM, SHORT_TREND.

Volatility Predictor XGBoost

Phase 3 | 7 models | File: volatility_model.py

Predicts future realized volatility for each timeframe. Uses historical volatility features (ATR, Parkinson volatility, Yang-Zhang estimator), VIX correlation, and intraday range patterns. Critical for dynamic position sizing — higher predicted volatility = smaller positions.

Exit Timing Model XGBoost

Phase 4 | 10 models | File: exit_timing_model.py

One per setup type. Predicts the optimal exit point (in bars) after entry. Trained on historical trades with labeled exit quality (early exit, optimal, late exit, stopped out). Features include time-in-trade, unrealized P&L trajectory, momentum decay, and volume patterns post-entry.

Sector-Relative Model XGBoost

Phase 5 | 3 models (daily, hourly, 5min) | File: sector_relative_model.py

Compares individual stock performance against its sector ETF. Features: relative strength ratio, correlation breakdown, sector rotation signals, and beta-adjusted momentum. Helps identify stocks outperforming/underperforming their sector — key for pair-style trades.

Gap Fill Probability Model XGBoost

Phase 5.5 | 7 models | File: gap_fill_model.py

Predicts the probability that a price gap will be filled within the session. Features: gap size relative to ATR, pre-market volume, overnight news sentiment, historical gap-fill rates per symbol. Critical input for GAP_AND_GO and SHORT_GAP_FADE setup decisions.

Risk-of-Ruin Model XGBoost

Phase 6 | 6 models (1min → daily) | File: risk_of_ruin_model.py

Estimates the probability of a catastrophic drawdown scenario per timeframe. Features: consecutive loss streaks, drawdown depth/duration, portfolio heat, correlation spikes, and volatility regime. Feeds the circuit breaker system and position sizing limits.

Regime-Conditional Models XGBoost

Phase 7 | 28 models (7 timeframes × 4 regimes) | File: regime_conditional_model.py

Rather than one model-fits-all, these are specialized models trained only on data from a specific market regime:

  • Bull Trend — trained on confirmed uptrend periods
  • Bear Trend — trained on confirmed downtrend periods
  • Range-Bound — trained on consolidation periods
  • High Volatility — trained on volatile/chaotic periods

At prediction time, the current regime is classified first, then the appropriate regime-specific model is selected. This dramatically improves accuracy vs. a single model that averages across all market conditions.

Ensemble Meta-Learner XGBoost

Phase 8 | 10 models (one per setup) | File: ensemble_model.py

The "model of models." Takes predictions from all Phase 1-7 models as input features and learns which models to trust in which conditions. Features: individual model predictions, confidence scores, agreement rates, current regime, volatility level. This is the final XGBoost layer that synthesizes all signals.

4.2 Deep Learning Models (PyTorch)

CNN Chart Pattern Recognition PyTorch CNN

Phase 9 | 13 models | Files: chart_pattern_cnn.py, chart_image_generator.py, cnn_training_pipeline.py

Converts OHLCV data into chart images and classifies visual patterns using convolutional neural networks. Identifies: flags, wedges, channels, triangles, head-and-shoulders, double tops/bottoms, cup-and-handle, and more. Each pattern gets a win probability score based on historical outcomes. 13 model variants cover different chart contexts and timeframes.

Architecture: Multi-layer CNN with batch normalization, dropout, and a final classification head that outputs both pattern type and win probability.

VAE Regime Detector PyTorch VAE

Phase 11 | 1 model | File: vae_regime.py

Variational Autoencoder that learns to compress market state into a low-dimensional latent space, then classifies the current market regime from that compressed representation.

Architecture:

  • Encoder: Linear(input→128) → ReLU → Linear(128→64) → ReLU → μ/σ heads
  • Latent Space: 16-dimensional with reparameterization trick
  • Decoder: Linear(16→64) → ReLU → Linear(64→128) → ReLU → Linear(128→input)
  • Regime Head: Linear(16→32) → ReLU → Linear(32→N_regimes)

Detected Regimes: bull_trending, bear_trending, high_volatility, mean_reverting, momentum_surge

Training: 100 epochs, batch size 256. Loss = reconstruction loss + KL divergence + regime classification cross-entropy.

Temporal Fusion Transformer (TFT) PyTorch Transformer

Phase 11 | 1 model | File: temporal_fusion_transformer.py

Multi-timeframe attention model that simultaneously processes data from multiple timeframes and learns which timeframe matters most for the current prediction.

Architecture:

  • Per-Timeframe Encoders: Linear(features→hidden) + ReLU + Dropout for each of 7 timeframes
  • Variable Selection Network: Learns importance weights per input variable
  • Temporal Self-Attention: MultiheadAttention (4 heads) with residual connections
  • Direction Head: Linear(hidden×7→hidden) → ReLU → Linear(hidden→3) [up/down/flat]
  • Confidence Head: Linear(hidden×7→hidden) → ReLU → Linear(hidden→1) → Sigmoid

Key Innovation: Outputs timeframe_weights showing which timeframe dominated the prediction (e.g., "15min: 0.4, daily: 0.35, 1hr: 0.25").

CNN-LSTM Temporal Model PyTorch CNN+LSTM

Phase 11 | 1 model | File: cnn_lstm_model.py

Hybrid architecture that uses CNN to extract spatial features from price windows and LSTM to capture temporal dependencies across those windows.

Architecture:

  • CNN Block: Conv1d(1→64, k=5) → BatchNorm → ReLU → Conv1d(64→128, k=3) → BatchNorm → ReLU → MaxPool → Flatten → Linear(128×16→256)
  • LSTM: 2-layer bidirectional LSTM (input=256, hidden=128) with dropout=0.2
  • Direction Head: Linear(128→64) → ReLU → Dropout → Linear(64→3) [up/down/flat]
  • Win Probability Head: Linear(128→64) → ReLU → Dropout → Linear(64→1) → Sigmoid
  • Confidence Head: Linear(128→64) → ReLU → Linear(64→1) → Sigmoid

Input: Rolling windows of 64 price bars per symbol. Processes temporal evolution of patterns.

5 The Confidence Gate — 11-Layer Additive Scoring

The confidence gate is the pre-trade intelligence layer. Every detected setup must pass through all 11 layers before a trading decision is made. The system uses additive scoring — starting from 0 points, each layer adds or subtracts confirmation points with floor protection (no single layer can veto a trade with more than -10 points).

Decision Thresholds
ScoreDecisionPosition SizeMeaning
≥ 55 ptsGOFull sizeStrong multi-signal confirmation. Execute with conviction.
≥ 30 ptsREDUCE60% sizeBorderline — some confirmation but not unanimous. Smaller position.
< 30 ptsSKIP0%Insufficient confirmation. Do not trade.

Maximum theoretical score: ~125 pts (all layers max). Maximum theoretical floor: ~-65 pts (all layers min).

Key Design Principle: Floor protection ensures no single layer can veto a trade. Even if a model strongly disagrees (-5), the other 10 layers can still confirm the trade. This prevents over-reliance on any single signal and rewards broad confirmation.

6 Live Trading Process

Here is exactly what happens when SentCom identifies and executes a trade:

1. Market Data Stream
IB Gateway streams live bars to the Spark backend via WebSocket. Data is normalized and stored in ib_historical_data. The enhanced scanner processes every incoming bar.
▼
2. Setup Detection
The EnhancedScanner evaluates 20 setup patterns (10 long, 10 short) on every bar update. When a pattern matches (e.g., VWAP reclaim on AAPL 5min), it generates a LiveAlert with entry price, stop price, targets, and quality score (0-100).
▼
3. SentCom AI Consultation (Optional)
High-quality alerts (A/A+ grade) can trigger a quick LLM consultation via the local Ollama model. SentCom's "brain" reviews the alert context, recent similar trades, current regime, and provides a natural language assessment.
▼
4. Confidence Gate Evaluation
The alert enters the 11-layer confidence gate. Each layer runs independently and contributes points. Total score determines: GO (≥55), REDUCE (≥30), or SKIP (<30). Full reasoning chain is logged.
▼
5. Position Sizing
For GO/REDUCE decisions, the PositionSizer calculates position size using:
• Account equity × risk percentage (usually 1-2%)
• Distance to stop loss
• Confidence gate's position_multiplier (0.0-1.5)
• Volatility prediction (Phase 3 model)
• Risk-of-ruin constraint (Phase 6 model)
• Circuit breaker limits (daily loss, consecutive losses, etc.)
▼
6. Order Execution
TradeExecution sends the order to IB Gateway via the Windows PC relay. Supports market, limit, and bracket orders. Stop-loss and take-profit orders are placed simultaneously. Execution quality (slippage, fill time) is tracked.
▼
7. Position Management
Active positions are monitored by the PositionManager. The Exit Timing model (Phase 4) provides guidance on when to close. Dynamic stop adjustments (trailing stops) are applied based on the Volatility model. Scale-out targets are calculated using the Advanced Targets module.
▼
8. Outcome Recording
When a position is closed (stop, target, or manual), the LearningLoopService records:
• Win/Loss and R-multiple
• Execution quality score
• Setup context (regime, quality, confidence score at entry)
• Actual vs predicted outcome
This data feeds back into the confidence gate (Layer 7) and future model training.
▼
9. Continuous Adaptation
Daily analysis runs automatically:
• Per-setup win rate tracking (edge decay detection)
• Trader profile updates (tilt detection, confidence calibration)
• Strategy promotion/demotion based on recent performance
• Trading mode adjustment: Aggressive → Normal → Cautious → Defensive
• Calibration recommendations generated for the next session

Trading Modes

ModeTriggerEffect
