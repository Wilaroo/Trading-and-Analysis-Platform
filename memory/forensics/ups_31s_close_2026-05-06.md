## TL;DR

A UPS trade fired the `oca_closed_externally_v19_31` sweep **31 seconds** after entry. The sweep's guard requires `age >= 30s` so this firing is on the absolute boundary — exactly the kind of timing where false positives can occur. This report documents the mechanism, lays out the legitimate-vs-false-positive heuristic, and ships a **re-runnable audit script** (`/app/backend/scripts/audit_ups_31s_close.py`) the operator can fire on the DGX where the data lives. ## 1. The mechanism

The sweep lives in `services/position_manager.py` (lines 175–312) and fires when **all three** conditions hold:

| Condition | Code |
|---|---|
| 1. Bot still tracks live shares | `_rem_external > 0` |
| 2. IB shows zero shares in BOTH directions for the symbol | `ib_qty_my_dir == 0 and ib_qty_opp_dir == 0` |
| 3. Trade age ≥ 30 seconds | `age_s_e >= 30` |

When all three hold, the trade is marked `CLOSED` with `close_reason="oca_closed_externally_v19_31"` and any IB realized PnL for that symbol is apportioned by share count and stamped onto `realized_pnl`.

The 30s grace is **the entire safety net** between false-positive sweeps and brand-new fills the IB pusher hasn't reported yet. A 31-second sweep is one second past the floor.

## 2. Two competing hypotheses

### Hypothesis A — Legitimate target hit (most likely)

The bracket's profit target genuinely fired at IB within 30s of entry. UPS at $97 with a 1-2% target only needs a $1-2 move; momentum-driven entries can easily clear that within 30 seconds on a fast intraday print.

**Evidence that supports A:**
- Non-zero `realized_pnl` on the trade row (target hits stamp realized PnL via the v19.31.12 apportionment).
- Matching IB execution tape rows (opposite-side fills) in `ib_executions` within ±60s of `closed_at`.
- The bracket's lifecycle events show a `target_filled` or `oca_target_hit` phase before sweep.

### Hypothesis B — False positive (the danger case)

The IB pusher snapshot was momentarily incomplete. After a fresh entry fill, the IB Gateway can take a few seconds to fully reconcile the position into its `positions` snapshot — the operator has hit this race twice already (v19.31 SOFI direction-flip, v19.34.15b UPS share-drift). If the pusher pushed a snapshot showing UPS=0 at the 30-31s mark, the bot would falsely sweep a still-live position.

**Evidence that supports B:**
- `realized_pnl == $0.00` on the trade (legitimate target hits would have stamped a positive number).
- No matching IB execution tape rows in the ±60s window — the broker never actually closed it.
- A subsequent v19.34.15b drift-loop run re-discovers UPS shares at IB after the bot's record was already swept.

## 3. The audit script

`/app/backend/scripts/audit_ups_31s_close.py` automates the classification using 5 weighted heuristics:

| Heuristic | Score impact |
|---|---|
| `realized_pnl > 0` (target likely hit) | +2 |
| `realized_pnl < 0` (stop likely hit) | +1 |
| `realized_pnl == $0.00` exactly | **−2** |
| `age` in [30, 35]s window | **−2** |
| `age > 60s` (well past floor) | +1 |
| Matching `ib_executions` opposite-side fill in ±60s | **+3** |
| Tape rows present but none on expected side | −1 |
| No `ib_executions` rows in window | **−2** |

**Verdict**: `score ≥ 3` → LEGITIMATE · `score ≤ −2` → SUSPICIOUS · else → UNKNOWN.

## 4. Recommended action by verdict

### If LEGITIMATE
No code change needed. The sweep is doing its job — closing out a stale bot record after a real target hit. Move on.

### If SUSPICIOUS
**Bump the age floor from 30s to 60s** for the `oca_closed_externally_v19_31` path. Rationale: real bracket target hits don't need to be cleaned up within 30s; the v19.34.15b drift loop will catch any genuine external close within its 30s tick anyway, and 60s gives the IB pusher far more margin to fully reconcile snapshots.

**Single-line patch** (`services/position_manager.py` line 217):
```diff
-                                    age_ok_e = age_s_e >= 30
+                                    age_ok_e = age_s_e >= 60
```

Plus add a defensive guard: **also require `realized_pnl != 0`** before sweeping. If IB's realizedPNL is still 0 for the symbol, the broker has not yet posted a close — do NOT sweep.

```diff
-                            if age_ok_e:
+                            # v19.34.16 — require both age >= 60s AND
+                            # IB-side realized PnL non-zero. Without
+                            # the realized check, the 30s sweep can
+                            # false-positive on pusher snapshot lag.
+                            if age_ok_e and ib_realized_for_sym != 0:
```

### If UNKNOWN
Ship the audit script as a recurring nightly job (cron) so the operator builds up a corpus of classifications. A few weeks of data will let us refine the heuristics into a hard verdict.
