### 1.3 🔬 DATA-TRUST FINDINGS — how raw data becomes the metrics that gate trades

All references are `services/realtime_technical_service.py` unless noted. These are
the calculations the operator distrusts; here is exactly what each does and where it
can lie. **Nothing changed yet — this is the diagnosis. Fix patches come after review.**

#### 🔴 F1 — RVOL is time-of-day NAÏVE (systematically understated intraday)
- **Code:** `daily_volume = today["volume"]` (:509); `avg_volume` = 20-day mean of
  **full-day** volumes (:512); `rvol = daily_volume / avg_volume` (:516).
- **Problem:** intraday, `today["volume"]` is the **partial cumulative** volume so
  far, but it's divided by a **full-day** average. At 10:00 ET a genuine 5×-volume
  mover can show `rvol ≈ 0.3`. RVOL is the single most-used gate in the scanner
  (`snapshot.rvol >= 1.5 / 1.8 / 2.0 / 2.5` appears in nearly every `_check_*`).
- **Impact:** the bot **under-fires liquid movers early in the session** and
  **over-fires late** (when cumulative volume naturally exceeds the daily avg).
  Directly distorts setup selection AND the TQS RVOL pillar (Phase 2 will confirm).
- **Correct approach:** time-of-day-adjusted RVOL — compare cumulative volume at
  minute-of-day *t* against the average cumulative volume at the **same** *t* across
  the lookback (a "volume curve"), OR use a same-elapsed-minutes intraday sum.

#### 🔴 F2 — Intraday indicator freshness is gated in DAYS, and bypassed by a live quote
- **Code:** staleness threshold is **trading days** (`max_trading_days = max(1,
  max_age_hours//24)`, default ⇒ **1 trading day**) (:255-311). A bar from *yesterday*
  returns `days_elapsed=1`, `1 > 1 == False` ⇒ **treated as FRESH**. Worse: when a
  live IB quote exists, the staleness check is **skipped entirely** (:442), and any
  intraday set with `len ≥ 5` is stamped `data_quality="real"` (:554).
- **Problem:** VWAP, RSI(14), EMA9/EMA20 — all computed from `intraday_bars` — can be
  built from **stale 5-min bars (up to a full trading day old, or skipped-stale under
  a live quote)** while the snapshot reads as real-time. The live quote only
  guarantees the **price** is fresh, not the **indicators** that gate the setups.
- **Impact:** `dist_from_vwap`, `dist_from_ema9`, `rsi_14` gates can fire/deny on
  indicators that don't reflect the current session. Highest risk at the open and
  after data gaps.
- **Correct approach:** a **minute-level** freshness gate on the trailing intraday
  bar (e.g. last 5-min bar must be ≤ N minutes old during RTH); if it fails, either
  refuse the scan or mark indicators `degraded` and block auto-exec on that symbol.

#### 🔴 F3 — Fabricated indicator FALLBACKS feed real stop/target geometry
- **Code (daily missing, :534-551):** `prev_close=price*0.99`, `high=price*1.01`,
  `low=price*0.99`, `avg_volume=1_000_000`, `rvol=1.0`, **`atr=price*0.02`**,
  `resistance=price*1.03`, `support=price*0.97`.
- **Code (intraday <5 bars, :575-582):** **`vwap=price*0.998`**, `ema9=price*0.99`,
  `ema20=price*0.985`, `rsi_14=50`.
- **Problem:** these are **invented numbers**, not measurements. They flow straight
  into setup logic that computes **real orders**: e.g. `stop_loss = vwap − atr*0.5`,
  `target = vwap + atr*1.5` (vwap_bounce, :4134-4135), `stop = resistance − atr`
  (breakout, :4262). A synthetic `atr=2% of price` + synthetic `vwap=price*0.998`
  produces a **fabricated risk/reward** the bot will trade on.
- **Impact:** worst-case for *unmanaged* trading — a symbol with thin/missing bars
  can still produce an alert with a made-up stop and target. RSI=50 fallback is the
  one safe default (fails the `rsi<38`/`>65` gates, so it tends to *not* fire).
- **Correct approach:** if the inputs needed for a setup's geometry are fallback/
  estimated (`data_quality != "real"`), **do not emit an auto-exec-eligible alert**.
  Surface as info-only at most. Never size a real bracket off an estimated ATR/VWAP.

#### 🟡 F4 — Snapshot cached 120 s; scanner reads without `force_refresh`
- **Code:** `_cache_ttl = 120` (:130); cache hit returns the old snapshot if
  `cache_age < 120` (:410-414). Scanner calls `get_technical_snapshot(symbol,
  mongo_only=True)` with **no `force_refresh`** (enhanced_scanner.py:3439).
- **Impact:** an auto-exec decision can run on a snapshot (price + RVOL + indicators)
  up to **2 minutes old**. For fast/breakout setups that's a meaningful slippage and
  can re-confirm a move that already reverted. (Note: the *entry* price is later
  re-anchored to live at order time per v283, which mitigates fill price — but the
  **decision** still used stale inputs.)

#### 🟡 F5 — VWAP / intraday EMAs are NOT session-anchored
- **Code:** `_get_intraday_bars_from_db` pulls the **last 78 5-min bars by recency**,
  no session/day boundary (:138-160); `_calculate_vwap` sums over whatever it's given
  (:750).
- **Impact:** pre-market, post-gap, or low-volume days, the 78-bar window blends the
  **prior session(s)** into "today's" VWAP and short EMAs → `dist_from_vwap` and
  `above_vwap` gates act on a multi-session VWAP, not the standard session-anchored
  VWAP traders (and the setups' thresholds) assume.

#### 🟡 F6 — RSI/ATR use simple averaging (not Wilder's); ATR includes the partial bar
- **Code:** `_calculate_rsi` simple mean of gains/losses (:816-823); `_calculate_atr`
  simple mean of TRs (:845-848) over **daily** bars where `bars[-1]` is the
  still-forming *today* bar.
- **Impact:** internally consistent (scanner == training, so ML isn't skewed), but
  thresholds borrowed from standard (Wilder) RSI/ATR conventions (`rsi<30`, ATR-based
  stops) are slightly miscalibrated. Lower priority than F1–F3, but worth a decision.

#### 🟢 F7 — `daily_bars[-1]` assumed to be "today" (verify pre-open)
- **Code:** `today = daily_bars[-1]` (:505) drives HOD/LOD/gap/volume.
- **Risk:** if the daily collector hasn't written today's 1-day bar yet (early pre-
  open), `today` is actually *yesterday* → gap %, HOD/LOD, RVOL numerator all shift a
  day. **To verify in Phase 1 follow-up / Phase 6** (depends on collector timing).

### 1.4 ✅ What EARNED trust in the scan/data layer

- **Skips are now counted + traced** (v281 `_symbols_skipped_no_data`,

### 4.6 🔬 FINDINGS

#### 🔴 M2 — Trailing/breakeven stops are LOCAL-ONLY between scale-outs (unmanaged risk)
Stop ratchets (`_update_trail_position`, `_move_stop_to_breakeven`, resnap) update only
`trailing_stop_config['current_stop']` in memory + `_record_stop_adjustment` (which just
logs + emits a stream event). **The resting IB bracket stop is re-written only at order
placement and after a scale-out (`reissue_bracket_for_trade`).** A pure trailing ratchet
(price extends, no target hit) does NOT modify the IB stop. So if the manage loop stalls,
the quote feed drops, or the process dies after the stop trailed up, **IB protects only at
the last-reissued stop (original or last scale-out level), not the latest trail** — the
trailed profit is unprotected server-side. This is the single highest-leverage gap for
**unmanaged** paper trading (the loop's liveness becomes the sole guardian of trailed
gains). **Proposal:** on a significant trail ratchet (> X% or > Y·ATR move since last
push, throttled), modify the IB stop order so the server-side bracket tracks the trail —
OR explicitly accept "IB backstop = original/scale-out stop" and require the operator to
monitor loop health. Needs an operator decision before any patch.

#### 🟡 M1 — Trailing engine can get STUCK at breakeven (never % -trails the runner)

```
elif 0 in targets_hit and mode == 'original': _move_stop_to_breakeven  # T1 → breakeven

```
On **sequential** fills: T1 hits → mode flips to `breakeven`. When T2 later hits, the
trailing branch's `mode == 'original'` guard is now False → `_activate_trailing_stop`
never runs, and `_update_trail_position` (needs `trailing`) never runs either. The runner
then ratchets ONLY via the 60s liquidity/anchor resnap — **the dedicated `trail_pct`
engine is bypassed**. Trailing only engages if T1+T2 land in the SAME scale-out tick
(gap-through). Mitigated (resnap still ratchets protectively), but the runner can give
back % -trail gains to breakeven. **Proposal:** allow `breakeven → trailing` transition
(activate when `1 in targets_hit and mode in ('original','breakeven')`). Operator
decision: should the post-T2 runner % -trail, or intentionally sit at breakeven + resnap?

## PHASE 6 — TRACK / LEARN  (report 2026-06-08)

**Scope:** the learning loop the unmanaged forward-test depends on — how a close
becomes a recorded outcome, how outcomes roll up into the EV/win-rate the gates
read, and what happens when a setup is data-starved. Verified against code.

### 6.1 The two outcome stores (verified)
There are **two distinct outcome collections**, fed on close:
```
close                                   ┌─→ alert_outcomes  (canonical EV/grade feed)
 │                                       │      → strategy_stats (EV gate, TQS Setup pillar)
 ├─ position_manager.close_trade ────────┤      → setup_grading / learning-loop rebuilds
 │   (3298 direct _record_alert_outcome) │
 │   (3337 learning_loop.record_outcome)─┼─→ trade_outcomes (ML confidence-gate calib + tilt)
 │                                       │
 └─ apply_close_pnl (silent paths) ──────┘─→ alert_outcomes ONLY
     operator-flatten / zombie / phantom-recovery / external-OCA
```
- **`alert_outcomes`** (pnl_compute) — feeds `strategy_stats` → the EV gate + TQS
  Setup pillar + grading + learning-loop rebuilds.
- **`trade_outcomes`** (learning_loop_service) — feeds the **AI confidence-gate
  auto-calibration** (`gate.record_trade_outcome`) + the **tilt state**.

### 6.2 ✅ What EARNED trust
- **Capture is HONEST (both stores).** R-multiple = `(exit−entry)/(entry−stop)`
  from **actual fills**, direction-aware — never projected R:R. `outcome`
  (won/lost/scratch) derives from realized PnL. (pnl_compute.py:149-163;
  learning_loop.py:347-351.)
- **`alert_outcomes` is idempotent + single-row-per-trade.** Upsert keyed on
  `trade_id` (pnl_compute.py:255). `close_trade` (3298) and the silent
  `apply_close_pnl` paths are mutually exclusive per trade, and even a double-fire
  overwrites the same row — **no double-count**.
- **EV rollups recompute WHOLE-TRADE** from `alert_outcomes`
  (`recompute_strategy_stats_for_setup`, pnl_compute.py:380). This replaced the
  v216 per-close-EVENT incremental counter that **double-counted scale-out
  partials** (accumulation_entry read 52%/+0.62R when realized was 11%/−0.43R).
  win_rate AND EV now share the SAME whole-trade sample and converge with the
  nightly `backfill_strategy_stats.py`. ✅
- **Genuine-only EV feed.** Non-genuine closes (phantom sweep / instant external
  unwind / operator flatten / corrupt entry==exit) are hygiene-tagged and
  **excluded** from `strategy_stats` (`if _genuine:`, pnl_compute.py:274), with a
  **legacy-row artifact guard** (v284) re-deriving genuineness from
  setup_type/close_reason for pre-v240 rows that lack the `genuine` flag.
- **Entry-time context join (the ML loop's spine).** `capture_alert_context`
  stamps regime/tape/grade at **signal** time (enhanced_scanner.py:8356,
  in `_process_new_alert`); `record_trade_outcome` joins it via
  `_pending_contexts.pop(alert_id)` at close — so the model learns on **entry**
  conditions, not exit conditions (v19.34.36 fixed the alert_id fallback that
  was corrupting this).
- **Data-starvation handled, not hidden.** setup_grading → `INSUFFICIENT_DATA`
  below 5 trades (surfaced, non-blocking); EV computed only when `len(r_out) ≥ 5`
  else `0.0`; the scanner cold-start grace (<20 outcomes → win-rate floor) +
  v294 0.33× size haircut cap unproven setups. `scripts/retrain_readiness.py`
  gates ML retrains on sample budget.

### 6.3 🔬 FINDINGS

#### 🟡 L1 — `trade_outcomes` (ML gate calibration) has NO genuine/artifact filter
`close_trade` fires `learning_loop.record_trade_outcome(...)` for **every** close
(position_manager.py:3334-3337) with **no `_genuine` guard** — the exact hygiene
filter the `alert_outcomes` EV feed applies (6.2) is **absent on this path**. So
**phantom-sweep / reconciled-orphan / external-OCA / operator-flatten closes leak
into the AI confidence-gate auto-calibration AND the tilt state.** This is the
same reconciliation-wreckage class that dragged the EV scoreboard before v240 —
but it still pollutes the **ML** learning loop, which is exactly what the
unmanaged forward-test leans on. **Highest-leverage Phase 6 fix.** **Proposal:**
gate the line-3337 `record_trade_outcome` call on the same `classify_close`
genuineness used by pnl_compute (or pass `genuine`/`hygiene_tag` through and let
the learning loop drop artifacts) — reversible via env.

#### 🟡 L2 — Store-coverage asymmetry: silent closes feed `alert_outcomes` but NOT `trade_outcomes`
`trade_outcomes` is fed **only** by `close_trade` (3337). The silent
`apply_close_pnl` paths (operator-flatten, zombie cleanup, phantom recovery,
external-OCA detection at :217/:2907/:3131/:3479) write `alert_outcomes` but
**never** call `learning_loop.record_trade_outcome`. Net: the ML confidence gate
+ tilt state **never learn from those closes at all** (under-coverage), while the
EV feed sees them. Combined with L1, the ML store has the **wrong** set: it
**includes** reconciled-orphan closes routed through close_trade but **excludes**
operator/zombie closes. **Decision needed:** make `trade_outcomes` coverage match
`alert_outcomes` (route silent genuine closes to it too) OR explicitly accept the
ML gate trains only on manage-loop/stop/target/EOD closes. (`learning_reconciler`
already deliberately touches `alert_outcomes` ONLY, by design — so the reconciler
is not the place to fix this.)

#### 🟢 L3 — Scratches (0-pnl AND 0-R) dropped from the win-rate denominator
`recompute._classify_outcome` returns `None` for a row with `outcome="scratch"`,
`r=0`, `pnl=0` → `continue` (pnl_compute.py:425) → excluded from `trig`. A true
breakeven is a real outcome; dropping it slightly **inflates win_rate** (removes
neither-win-nor-loss from the denominator). Matches `backfill_strategy_stats`, so
train/serve-consistent — note only.

#### 🟢 L4 — `avg_loss_r` defaults to 1.0 when a setup has zero losses
pnl_compute.py:439 — a setup with only wins (≥5 R-outcomes) computes EV against a
**synthetic** 1R loss magnitude. Conservative (caps EV), but it's an invented
number on a thin-data setup. Note only.

#### 🟢 P2-LEGACY (re-confirmed, full detail) — `enhanced_scanner.record_alert_outcome`
Reached **only** by the manual API route `routers/live_scanner.py:330`
(`POST /api/scanner/stats/record-outcome`) — **no autonomous caller**. But it is a
genuinely dishonest writer: **non-idempotent `insert_one`** (enhanced_scanner.py:2354,
duplicates a trade pnl_compute already upserted), **wrong base key**
`split("_")[0]` (2296 — `gap_give_go`→`gap`), **projected-R win fallback**
(2328 — `r=alert.risk_reward` when realized R≤0), **−1R loss cap** (2337), and a
**double `_calculate_expected_value`** (record_r_outcome + update_win_rate)
corrupting `ev_trend`. The wrong key INSULATES multi-word setups (their canonical
base never matches the truncated key), but **single-word setups (`squeeze`,
`orb`, `hitchhiker`) collide** → manual calls pollute their canonical EV with
projected-R/−1R-capped rows. → **Phase 7: fix-or-delete the route + method.**
